# Database Sharding

We've talked about scaling the web server layer using Load Balancers, and scaling the read layer using Caching. But what happens when your core relational database simply gets too big? 

If you have 10 Terabytes of user data, and the biggest hard drive you can afford is 2 Terabytes, you have a physical limitation. You can no longer **Scale Up** (Vertical Scaling). You must **Scale Out** (Horizontal Scaling).

For databases, scaling out is called **Sharding**.

## What is Sharding?

Sharding is the process of breaking up a single massive database into smaller, more manageable pieces, called **Shards**. Each shard is an independent database, hosted on its own separate server, containing a specific slice of the overall data.

If you have 10 million users, you might put:
- Users 1 to 3,000,000 on Server A (Shard 1)
- Users 3,000,001 to 6,000,000 on Server B (Shard 2)
- Users 6,000,001 to 10,000,000 on Server C (Shard 3)

## The Shard Key

The most critical decision in a sharded architecture is choosing the **Shard Key** (or Partition Key). This is the piece of data used to determine which server a specific row belongs to.

### Common Sharding Strategies

1. **Range Based Sharding:** 
   Splitting data based on ranges of a value (e.g., User ID 1-1000, 1001-2000). 
   *Problem:* If new users are much more active than old users, your newest shard will get hammered with traffic while the others sit idle.

2. **Hash Based Sharding:**
   Applying a mathematical hash function to the Shard Key (e.g., `hash(user_id) % 4`) to distribute data randomly but deterministically across 4 servers.
   *Problem:* If you need to add a 5th server, the math changes (`% 5`), meaning almost every piece of data has to be moved to a different server.

3. **Consistent Hashing:**
   An advanced hashing technique that solves the problem above. It maps data to a conceptual "ring" so that adding or removing a server only requires moving a small fraction of the data.

## The Cost of Sharding

Sharding is considered an absolute last resort in System Design. Why?
- **No Joins Across Shards:** You cannot easily perform a SQL `JOIN` if the Users table is on Server A, and their Payments table is on Server B.
- **Complex Logic:** Your application code (or a routing proxy) now has to be smart enough to know which server to ask for data.
- **Resharding:** If a single shard gets too big (a "hot shard"), splitting it again is an operational nightmare.

Always exhaust Vertical Scaling, Read Replicas, and Caching before you attempt to Shard your database!
